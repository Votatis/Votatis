# 스펙 리뷰하기 (spec-review)

구현 결과를 스펙 기준으로 검토한다. 코드 품질이 아니라 "스펙을 충족했는가"가 1차 기준이다.

## 작업 순서

1. **대상 확정** — 검토할 스펙(보통 `specs/in-progress/<NNNN>-<slug>.md`)과 대응하는 변경(브랜치/디프/파일)을 확정한다.
2. **완료 조건 대조** — Acceptance Criteria 각 항목을 코드/동작과 1:1로 대조해 충족/미충족/불명확으로 판정한다.
3. **요구사항·비목표 확인** — 요구사항 누락, 비목표 침범(범위 초과)이 없는지 본다.
4. **검증** — 가능하면 테스트 실행·동작 확인으로 판정 근거를 만든다.
5. **리포트** — 아래 형식으로 보고한다.
6. **QA 모드 진입** — 리포트를 낸 뒤 곧바로 **QA 모드**로 들어간다. 아래 "QA 모드" 절차를 따른다.
7. **상태 처리** — QA 모드가 끝나면, 통과면 파일을 `specs/completed/`로 옮기고 `status: completed`로 갱신한다. 보완 필요면 `in-progress`에 두고 [[spec-implement]]로 이어간다. 상태가 바뀌면 `specs/README.md` 인덱스의 해당 행(상태·갱신일·파일 경로)도 맞춘다.
8. **커밋** — QA 모드 종료 이후에만 변경을 커밋한다. 아래 "커밋" 절차를 따른다.

## QA 모드

리뷰 리포트를 낸 직후 QA 모드에 진입한다. QA 모드는 사용자와 주고받으며 변경사항을 처리하는 대화 루프다.

1. **진입 안내** — "QA 모드에 진입했습니다. 변경사항을 입력해 주세요. 끝나면 `종료` 또는 `끝`이라고 입력하세요."라고 알린다.
2. **Queue 적재** — 사용자가 변경사항을 입력하면, 즉시 처리하지 말고 **Queue에 항목으로 쌓는다.** 한 번에 여러 건을 입력하면 각각을 별도 항목으로 분리해 넣는다. 적재 후 현재 Queue 상태(대기 중 항목 목록)를 짧게 보여 준다.
3. **하나씩 처리** — Queue에 쌓인 항목을 **서브에이전트(Agent 도구)로 하나씩 순차 처리**한다. 항목 하나당 서브에이전트 하나를 띄워 그 변경을 구현/적용하게 하고, 결과를 받아 사용자에게 보고한 뒤 다음 항목으로 넘어간다. 한 항목이 끝나기 전에 다음 항목을 시작하지 않는다.
4. **추가 입력 수용** — 처리 도중 사용자가 새 변경사항을 입력하면 그것도 Queue 뒤에 적재한다. Queue가 빌 때까지 처리를 계속한다.
5. **스펙에 반영** — QA 모드 중에 처리한 변경이 **기능적(동작 추가/변경/제거)이거나 기술적(스택·구조·스키마 변경)**이면, 코드만 고치고 끝내지 말고 **대응 스펙 문서에도 반영**한다.
   - 동작·요구사항·완료 조건이 바뀌었으면 스펙 본문(요구사항/Acceptance Criteria 등)을 수정한다.
   - 스펙 `## Changelog`에 한 줄 기록하고 frontmatter `updated`를 갱신한다(형식: `- YYYY-MM-DD: <무엇이 어떻게 바뀌었나> (요청: 채팅)`).
   - 단순 버그·오타·문구 수정이나 동작이 그대로인 리팩터링은 스펙에 기록하지 않는다.
6. **종료** — 사용자가 `종료` 또는 `끝`이라고 입력하면 QA 모드를 끝낸다. 남은 Queue 항목이 있으면 처리할지 사용자에게 확인한 뒤 종료한다.
7. **종료 후** — QA 모드가 끝나야 비로소 "상태 처리"와 "커밋" 단계로 넘어간다. **QA 모드가 끝나기 전에는 커밋하지 않는다.**

> QA 모드 중 발생한 기능적·기술적 변경은 코드뿐 아니라 스펙(본문 + Changelog)에도 반영한 뒤 닫는다.
> QA 모드 중에는 절대 커밋하지 않는다. 커밋은 사용자가 `종료`/`끝`으로 QA 모드를 명시적으로 닫은 뒤에만 이뤄진다.

## 커밋

리뷰 결과를 확정하면 관련 변경(코드 + 스펙 파일 이동/갱신)을 함께 커밋한다.

- **커밋 메시지에 스펙 파일 이름을 반드시 포함**한다. 확장자 제외한 파일명(예: `0001-report-intake-api`)을 쓴다.
- 형식: `<요약> (spec: <NNNN>-<slug>)`
  예) `제보 접수 API 구현 완료 (spec: 0001-report-intake-api)`
- 스펙 파일의 디렉터리 이동은 가능하면 `git mv`로 한 뒤 같은 커밋에 포함해 히스토리를 보존한다.
- 현재 기본 브랜치(main)에 있으면 먼저 작업 브랜치를 만들고 커밋한다.
- 커밋만 한다. push·PR은 사용자가 요청할 때만.

## 리포트 형식

```markdown
## 스펙 리뷰: <스펙 제목>

### 완료 조건 대조
- [x] <조건> — 충족 (근거: 파일:라인 / 테스트)
- [ ] <조건> — 미충족 (이유)
- [?] <조건> — 불명확 (확인 필요)

### 요구사항 누락 / 범위 초과
- ...

### 판정
통과 | 보완 필요 | 미통과
```

## 원칙

- 판정에는 근거(파일:라인, 테스트 결과)를 붙인다. 인상비평 금지.
- 미충족 항목은 무엇을 어떻게 고쳐야 하는지까지 적는다 — [[spec-implement]]로 바로 이어질 수 있게.
- 스펙 자체가 잘못됐다고 판단되면 그 점을 별도로 지적한다.
