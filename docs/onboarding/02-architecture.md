# 2. 시스템 구조 — 제보가 들어와서 공개되기까지

> 비유: **도서관**을 생각하세요.
> 누가 책(제보)을 기증함 → 사서가 진짜인지 검수 → 통과한 책만 서가(공개 아카이브)에 꽂힘.
> 검수 안 된 책은 절대 서가에 안 올라갑니다.

## 전체 흐름 한 장

```
 ┌──────────┐   1. 제보 작성        ┌─────────────────┐
 │  제보자   │ ───────────────────▶ │  frontend        │
 │ (브라우저)│   (Turnstile 통과,   │  (Next.js, 제보  │
 └──────────┘    EXIF 추출)         │   4단계 마법사)  │
                                     └────────┬────────┘
                       2. 첨부 업로드          │ 3. 제보 본문 전송
                          (presigned PUT)      │
                              ▼                ▼
                       ┌──────────┐    ┌─────────────────┐
                       │   R2     │◀───│  intake-api      │
                       │ (이미지) │    │  (Cloudflare     │
                       └──────────┘    │   Worker)        │
                                        │  정규화+해시봉인 │
                                        └────────┬────────┘
                                                 │ 4. 저장
                                                 ▼
                                        ┌─────────────────┐
                                        │  D1 (SQLite)     │
                                        │  status:         │
                                        │  pending →       │
                                        │  unverified      │  ← 검증 대기 큐
                                        └────────┬────────┘
                                                 │ 5. 사람이 검증
                          ┌──────────────────────┴───────────────────┐
                          ▼                                            ▼
                 ┌─────────────────┐                         ┌─────────────────┐
                 │ 관리자 페이지    │                         │  admin-mcp       │
                 │ (frontend/admin) │   둘 다 같은            │  (AI가 검증 보조)│
                 │  사람이 클릭     │   관리자 API 호출       │                 │
                 └────────┬────────┘                         └────────┬────────┘
                          └──────────────────┬───────────────────────┘
                                             ▼ 6. 판정 기록 (status 변경)
                                   confirmed / suspected / disputed /
                                   debunked / corrected  (= 공개 대상)
                                             │
                            ┌────────────────┴────────────────┐
                            ▼                                  ▼
                   ┌─────────────────┐                ┌─────────────────┐
                   │ 공개 아카이브    │                │ GitHub 마크다운  │
                   │ (frontend, SSG  │                │ export (공개     │
                   │  + 공개 API)    │                │  데이터 배포)    │
                   └─────────────────┘                └─────────────────┘
```

## 단계별로 천천히

### 1) 제보 작성 (frontend)
제보자가 웹사이트의 **4단계 마법사**로 제보를 씁니다. 봇이 아닌지 **Turnstile**로
확인하고, 사진을 올리면 브라우저에서 **EXIF**(촬영 정보)를 미리 읽습니다.

### 2~3) 업로드 & 전송 (frontend → R2 / intake-api)
- 첨부 이미지는 **presigned URL**로 **R2에 직접 업로드**합니다(서버를 거치지 않아 빠름).
- 제보 본문은 **intake-api**로 전송됩니다.

### 4) 저장 & 봉인 (intake-api → D1)
intake-api가 데이터를 **정규화**하고, 첨부는 **SHA-256 해시로 봉인**합니다
(나중에 변조 여부를 누구나 확인 가능). 그리고 **D1**에 저장합니다.
- 처음엔 `pending`(업로드 미완료, 내부 상태) → 완료되면 `unverified`(검증 대기).

### 5~6) 검증 (관리자 페이지 또는 admin-mcp)
사람이 **검토 큐**(`unverified`)를 보고 출처·첨부를 확인한 뒤 **판정**합니다.
판정하면 `status`가 바뀝니다:

| status | 한글 | 뜻 |
|--------|------|-----|
| `unverified` | 미검증 | 아직 안 봄 (기본) |
| `reviewing` | 검토 중 | 보는 중 |
| `confirmed` | 사실확인 | 확인됨 (단, 부정선거 단정 금지 — 범위 제한 필수) |
| `suspected` | 의심 | 통상 설명으로 해소 안 되는 미해명 정황 (조작 단정 아님) |
| `disputed` | 이견 있음 | 주장에 반대 근거가 있음 |
| `debunked` | 반박됨 | 거짓으로 확인 |
| `corrected` | 정정됨 | 일부 사실+정정 |

> 핵심 규율(**페르소나5**): 근거 없는 판정 금지, 주장과 사실 분리, 과잉해석 차단.
> 그래서 `confirmed/suspected/disputed/debunked/corrected` 판정에는
> **검증 방법·근거 링크·공개 요약** 등이 반드시 있어야 저장됩니다(없으면 거부).

### 7) 공개
검증을 통과(=공개 대상 상태)한 레코드만 **공개 아카이브**에 노출되고,
**GitHub 마크다운으로 export**되어 공개 데이터로 배포됩니다.
- 공개 아카이브 frontend는 **빌드타임 인덱스**(미리 구운 목록) + **런타임 공개 API**를 함께 씁니다.
- 증거 이미지는 R2 public 도메인에서 직접 보거나(설정 시) Worker 스트리밍으로 봅니다
  ([steering/evidence-image-public-serving](../../steering/evidence-image-public-serving.md)).

## 왜 이렇게 나눴나
- **검증 우선**: 미검증과 사실확인을 절대 같은 무게로 안 보여줍니다(PRD 원칙 1).
- **무결성**: 첨부는 해시로 봉인해 사후 변조를 막습니다(원칙 3).
- **단일 출처**: 검증 규칙은 intake-api에만 있고, 관리자 페이지·admin-mcp는 그걸 호출만 합니다.
