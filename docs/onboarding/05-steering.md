# 5. steering — 프로젝트의 "실수 노트"

같은 실수를 두 번 하지 않으려고, 작업하며 배운 교훈(규칙·함정·결정 이유)을
한곳에 쌓아 두는 곳이 `steering/`입니다.

## steering이 뭐고 왜 있나

`steering/` 폴더는 작업하면서 쌓인 **규칙**, **반복하면 안 되는 실수**, **비자명한 결정의 이유**를
누적하는 곳입니다. **파일 하나 = 교훈 하나**예요.

왜 필요하냐면:
- 코드만 봐서는 "왜 이렇게 했는지", "여기서 뭘 조심해야 하는지"를 알 수 없습니다.
- 예: "테스트는 초록인데 브라우저에서만 깨지는 함정" 같은 건 코드에 안 적혀 있죠.
- 그런 **눈에 안 보이는 지뢰**를 글로 남겨, 다음 사람(과 AI)이 안 밟게 합니다.

## 어떻게 생겼나

각 파일은 frontmatter + 본문입니다:

```markdown
---
tldr: 한 줄 요약 (세션 시작마다 주입되니 짧게!)
tags: [pitfall, cors, frontend]
last_retrieved: 2026-06-20   # 마지막으로 참조한 날
retrieval_count: 9           # 몇 번 써먹었나
---

## 규칙 / 교훈
...무엇을 / 왜 / 어떻게 적용하나...
```

[`steering/README.md`](../../steering/README.md)는 **인덱스(목차)**입니다.
실제 정책(운영 규칙)은 루트 [`CLAUDE.md`](../../CLAUDE.md)의 `## Steering` 절에 있어요.

> 💡 토큰 절약: 세션이 시작될 때 모든 파일이 아니라 **인덱스의 `파일 — tldr`만** 주입됩니다.
> 그래서 tldr는 짧고 핵심만 담아야 합니다.

## 4가지 핵심 동작

1. **회상 (작업 전):** 작업 시작 전에 인덱스를 훑어, 지금 일과 관련된 항목이 있으면
   그 파일을 읽고 따릅니다.
2. **자동 캡처 (작업 중):** ①합의된 규칙 ②저지른 실수+회피법 ③비자명한 결정의 이유가
   나오면 `steering/<kebab-case>.md`로 저장합니다.
   - 단, **코드·git·스펙으로 알 수 있는 사실이나 일회성 정보는 저장 안 함.**
3. **회수 통계 갱신:** 어떤 항목을 실제로 참조·적용했으면 그 파일의 `retrieval_count`를 +1,
   `last_retrieved`를 오늘로 갱신하고, 인덱스 표의 해당 행도 맞춥니다.
4. **인덱스 일관성:** 파일을 추가/수정/삭제하면 `steering/README.md` 표를 즉시 맞춥니다.

## 실제 예시 (지금 있는 것들 중)

steering을 읽으면 "아, 이걸 조심하라는 거구나"가 바로 옵니다:

- **web-dev-port-cors** — Next dev 포트(3000)와 API의 ALLOWED_ORIGIN이 안 맞으면
  "API는 200인데 브라우저만 실패". 포트를 맞춰라.
- **cors-allow-methods-new-verb** — 새 HTTP 메서드 추가 시 preflight 허용 목록도 같이 고쳐라.
  안 하면 vitest는 통과(직접 호출이라)하는데 브라우저만 깨진다.
- **react-ime-composing-enter** — 한글 IME 조합 중 Enter는 조합 확정용. 처리 안 하면
  "내손2동동"처럼 글자가 겹친다.
- **wrangler-deploy-env** — 배포 자격증명은 `CLOUDFLARE_*`(CF_* 아님), `scripts/cf-env.sh`로 매핑.

> 패턴이 보이죠? 대부분 **"이렇게 하면 조용히 깨진다"**는 함정 기록입니다.

## 언제 내가 steering을 건드리나

- 새 규칙에 합의했을 때 / 실수를 하고 회피법을 알았을 때 / 비자명한 결정을 내렸을 때 → **새 파일 작성**.
- 기존 교훈을 실제로 써먹었을 때 → **회수 통계 +1**.
- "이거 기억해둬", "다음부턴 ~하지 마" 같은 요청을 받았을 때.

스킬 `/steering`이 이 과정을 도와줍니다.
