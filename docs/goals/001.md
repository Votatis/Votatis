# Goal 001 — MVP-PRD 전 범위 완성

## 한 줄 목표

스펙 주도 개발 스킬을 활용해 `docs/MVP-PRD.md`에 정의된 **모든 내용**을 구현·완성한다. 인간의 개입이 무조건 필요한 일은 `loops/HUMAN.md`에 적어두고, **네가 스스로 할 수 있는 것부터** 진행한다.

## 진실 공급원 (먼저 읽어라)

- `docs/MVP-PRD.md` — 완성해야 할 전체 범위. 이 문서의 §3 v1 범위, §5 수집 흐름, §6 검증 워크플로우, §8 폼↔스키마, §9 검색, §10 구현 현황, §11 웹 기획이 작업 목록의 근거다.
- `docs/PERSONA.md` — 설득 대상(판단 유보자·회의적 비신자 등). UI·카피·검증 표기 결정 시 이 페르소나의 신뢰를 깎지 않는지 항상 점검한다.
- `CLAUDE.md` — 모노레포 구조, 스펙 주도 개발, steering, Changelog 정책. 반드시 준수.
- `specs/README.md` — 기존 스펙 인덱스와 상태. 이미 끝난 것/진행 중인 것을 먼저 파악한다.

## 작업 방식 (필수)

1. **현황 파악**: `docs/MVP-PRD.md`의 항목과 `specs/{completed,in-progress,in-review,not-started}` 현황을 대조해, **아직 안 된 것**의 목록을 만든다. 이미 구현된 것을 다시 만들지 않는다.
2. **스펙 주도**: 새 기능은 임의로 코딩하지 말고 스펙으로 시작한다.
   - `spec-create`로 `specs/not-started/`에 스펙 작성 → `spec-implement`로 구현(상태를 in-progress→in-review로 이동) → `spec-review`로 검증 후 `completed` 이동.
   - 스펙 번호·파일 이동·`specs/README.md` 인덱스 동기화는 각 스킬 규칙을 따른다.
3. **모노레포 규칙**: 새 앱/서비스는 `apps/<name>/`에 둔다(루트 금지). 전체 검증은 `pnpm -r test` / `pnpm -r typecheck`, 개별은 `pnpm --filter <pkg> ...`.
4. **steering 회상·캡처**: 작업 전 `steering/README.md` 인덱스를 훑어 관련 규칙·함정을 적용하고, 새 규칙/실수/비자명한 결정이 나오면 캡처한다.
5. **Changelog**: 기능·구조가 바뀌면 관련 스펙의 `## Changelog`에 한 줄 기록한다.

## 의사결정 원칙 (가장 중요)

**모든 제품·기술·설계 결정은 네가 직접 내린다.** 사람에게 "이거 어떻게 할까요?"라고 방향을 되묻지 않는다. 다만 모든 결정의 판단 기준은 **단 하나**다:

> **"이 결정이 `docs/PERSONA.md`의 사용자에게 유익한가?"**

- 선택지가 둘 이상일 때(UI 구조, 카피 톤, 검증 표기 방식, 기능 우선순위, 데이터 노출 범위 등) → PERSONA의 핵심 대상인 **판단 유보자(김도현)**, **회의적 비신자(이수진)**가 봤을 때 **신뢰가 올라가고 유익하다고 느낄** 쪽을 고른다.
- 절제·중립·출처 투명성처럼 PERSONA의 신뢰를 지키는 선택을, 화려함·선동·편의보다 우선한다.
- 결정을 내릴 때 그 근거(어느 페르소나에게 왜 유익한지)를 스펙/커밋/Changelog에 한 줄로 남겨, 결정이 추적 가능하게 한다.
- 단, 사람의 **권한·자원·법적 책임**이 물리적으로 필요한 일(아래 `loops/HUMAN.md` 항목)은 결정의 영역이 아니라 위임의 영역이다 — 그건 적어두고 넘어간다.

## UI 재사용 원칙 (필수)

새 화면을 만들 때 **기존 UI를 최대한 재사용**한다. 새 룩앤필을 발명하지 않는다.

1. **먼저 찾아서 재사용**: 구현 전 `apps/frontend`의 기존 컴포넌트·스타일·디자인 토큰(색상, 타이포, 간격, 라운드, 그림자 등)과 공통 CSS(예: `app-shell.css`)를 조사한다. 같은 목적의 컴포넌트가 있으면 그대로 쓰거나 확장한다.
2. **없으면 그때만 신규**: 기존에 없는 UI만 새로 만든다. 단 새로 만들 때도 **기존 룩앤필을 최대한 준수**한다 — 기존 토큰·간격 체계·컴포넌트 패턴을 따르고, 임의의 새 색/폰트/스타일을 도입하지 않는다.
3. **일관성 우선**: 기존 페이지(랜딩, 제보 마법사, 검색/아카이브 등)와 시각적으로 이질감이 없어야 한다. 새 화면이 "다른 사이트처럼" 보이면 PERSONA 1·2의 신뢰를 깎는다(절제·일관성 원칙).
4. 재사용/신규 결정을 스펙에 한 줄로 남긴다("기존 X 재사용" / "신규 Y — 토큰 Z 준수").

## UX 기획 → 8년차 UX 디자이너 리뷰 (필수)

화면이 있는 작업은 코드부터 짜지 말고 **UX를 먼저 기획**하고, 그 기획을 **8년차 UX 디자이너의 시선으로 셀프 리뷰**해 보완한 뒤 구현한다. 스펙(`spec-create`) 안에 이 과정을 담는다.

1. **UX 기획**: 사용자 플로우, 화면 구조(정보 위계), 주요 상태(빈 상태/로딩/에러/성공), 입력·검증 흐름, 핵심 인터랙션을 글/간단한 와이어프레임으로 정의한다. 어느 페르소나의 어떤 과업을 위한 화면인지 명시한다.
2. **8년차 UX 디자이너 리뷰**: 다음 관점으로 기획을 비판적으로 점검하고 발견한 문제를 보완한다.
   - 정보 위계와 시각적 우선순위가 사용자 과업과 맞는가
   - 인지 부하: 한 화면이 과하지 않은가, 단계가 적절히 분해됐는가
   - 일관성: 기존 UI 패턴·용어·컴포넌트와 일치하는가(위 UI 재사용 원칙)
   - 접근성: 대비, 키보드/포커스, 폼 레이블, 에러 메시지 명확성
   - 엣지/상태: 빈 상태·로딩·에러·권한 없음 등 누락된 상태가 없는가
   - 신뢰 신호: PERSONA 기준 — 검증 상태 라벨/출처/반박이 적절히 노출되어 절제·중립이 지켜지는가
   - 마이크로카피: 문구가 선동적이지 않고 명확·중립적인가
3. **반영 후 구현**: 리뷰에서 나온 개선점을 기획에 반영한 다음에 구현에 들어간다. 리뷰 결론(주요 발견·결정)을 스펙에 요약으로 남긴다.

> 관리자 페이지처럼 내부 화면은 **페르소나 5(정한울)** 기준으로 리뷰한다 — 빠른 통과가 아니라 엄격한 판정을 쉽게 만드는 UX인가.

## 우선순위 정책

1. **네가 코드로 완결할 수 있는 것 먼저.** 외부 계정·시크릿·결제·법률 판단·디자인 승인 없이 끝낼 수 있는 작업을 최우선으로 진행한다.
2. MVP-PRD §10에서 "개발 중(v1)"으로 표시된 항목(관리자 페이지, AI 분석 백엔드, D1→GitHub 마크다운 변환, 정적 사이트)을 핵심 타깃으로 삼는다.
3. 한 항목을 끝까지(스펙→구현→검증→Changelog) 완료한 뒤 다음으로 넘어간다. 여러 개를 미완성으로 벌려두지 않는다.

## 인간 개입이 필요한 일 → `loops/HUMAN.md`

다음처럼 **너 혼자 끝낼 수 없는 것**은 직접 시도하지 말고 `loops/HUMAN.md`에 항목으로 적어둔 뒤, 가능한 다른 작업을 계속한다. (파일/디렉터리가 없으면 생성한다.)

- 외부 서비스 계정·시크릿·바인딩 발급 (Cloudflare 토큰, R2/D1 운영 바인딩, Turnstile 운영 sitekey/secret, GitHub 배포 토큰 등)
- 운영 배포 승인 (Pages/Worker production deploy, 도메인 연결)
- 법률·정책 판단 (저작권·초상권·개인정보, 라이선스 확정, 마스킹 규칙 승인)
- 관리자 인증·권한 **부여 실행** (누구에게 실제로 검증 권한 계정을 줄지 — 정책·구조 설계는 네가 PERSONA 기준으로 결정)

> 제품 방향성·UI·카피·기능 우선순위 같은 **설계 판단은 사람에게 넘기지 않는다.** 위 의사결정 원칙에 따라 PERSONA 기준으로 네가 직접 정한다. 여기에 적는 것은 오직 사람의 권한·자원·법적 책임이 물리적으로 필요한 일뿐이다.

`loops/HUMAN.md` 항목 형식(권장):

```markdown
## [열림] <한 줄 제목>
- 무엇이/왜 필요한가:
- 막힌 작업(블로킹 대상):
- 사람이 해줘야 할 구체적 액션:
- (해결되면) 이어서 진행할 것:
```

해결된 항목은 `[닫힘]`으로 표시하고 처리 결과를 남긴다.

## 완료 정의 (Definition of Done)

- MVP-PRD §3 v1 범위의 모든 항목이 스펙으로 추적되고, 구현 가능한 것은 `completed` 상태다.
- 구현 코드는 `pnpm -r typecheck`와 `pnpm -r test`를 통과한다.
- 사람 개입이 필요해 미완으로 남는 것은 전부 `loops/HUMAN.md`에 명시돼 있다(조용히 빠뜨리지 않는다).
- 각 변경은 관련 스펙 Changelog와 steering(해당 시)에 반영돼 있다.

## 매 실행 시 루틴 (이 파일을 /goal로 다시 먹였을 때)

1. 진실 공급원과 `specs/README.md`, `loops/HUMAN.md`를 읽어 현재 상태를 파악한다.
2. 남은 작업 중 "네가 스스로 할 수 있는 가장 우선순위 높은 것" 하나를 고른다.
3. 스펙 주도 흐름으로 그 항목을 끝까지 완료한다.
4. 진행 중 막히면 그 사유를 `loops/HUMAN.md`에 적고 다음 가능한 작업으로 넘어간다.
5. 더 이상 사람 개입 없이 진행할 수 있는 작업이 없으면, 남은 `loops/HUMAN.md` 항목을 요약 보고하고 멈춘다.
