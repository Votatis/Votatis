# Goal 003 — 관리자 페이지 / 제보 UX 개선 묶음

## 의사결정 원칙 (가장 중요)

**모든 제품·기술·설계 결정은 네가 직접 내린다.** 사람에게 방향을 되묻지 않는다. 판단 기준은 하나:

> **"이 결정이 `docs/PERSONA.md`의 사용자, 특히 내부 검증 관리자(페르소나 5)에게 유익한가?"**

- 도구를 무엇을 노출/숨길지, 가드레일을 어디까지 둘지, 도구 description을 어떻게 쓸지 등은 — **검증자가 더 엄격하고 일관되게(확증편향 없이) 판정하도록 돕는** 쪽으로 정한다.
- AI 보조 신호는 "참고"로만 제시하고 자동 판정하지 않는다(페르소나 5 규율).
- 결정 근거(어느 페르소나에게 왜 유익한지)를 스펙/커밋/Changelog에 한 줄로 남긴다.

## 진실 공급원 (먼저 읽어라)

- `docs/PERSONA.md` — 특히 **페르소나 5(내부 검증 관리자)**.
- `docs/MVP-PRD.md` — 제보 카테고리, 검증 워크플로우, 관리자 페이지 명세.
- 관리자 인증 스펙 `specs/completed/0015-admin-auth-accounts.md` 및 관련 코드(JWT access/refresh, 회원관리, 로그인 화면).
- 프런트엔드 제보 카테고리 정의(투개표 / 사전투표 / 전산집계)가 있는 곳.
- `CLAUDE.md` — 모노레포, 스펙 주도 개발, steering, Changelog 정책.

## 할 일 (Definition of Done)

- [ ] **1. 제보(report) 카테고리 정합성** — 현재 프런트엔드 제보 카테고리는 **투개표 / 사전투표 / 전산집계** 3종이다. 이 카테고리가 report(제보 수집·저장·관리자 목록/필터)에도 동일하게 존재·반영되도록 맞춘다.
- [ ] **2. 관리자 세션 유지** — `/admin` 진입 시 이미 로그인했는데도 자꾸 다시 로그인을 요구하는 문제를 고친다. 한 번 로그인하면 세션이 유지되어야 한다(토큰 저장·갱신 흐름 점검).
- [ ] **3. 로그인 화면 안내 문구 제거** — 다음 설명 텍스트를 화면에서 없앤다:
  > 계정은 루트 관리자가 발급합니다. 비밀번호는 발급받은 재설정 링크에서 직접 설정하세요.
  > 세션은 이 브라우저에만 저장되며 요청 시 인증 헤더로 전송됩니다.
- [ ] **4. 관리자 페이지에서 제보 입력** — 관리자 페이지에서도 제보를 직접 집어넣을(생성) 수 있어야 한다.
- [ ] **5. 화면 미리보기 링크 제거** — 관리자 페이지 헤더/사이드바에 있는 "화면 미리보기"로 연결되는 링크를 없앤다.
- [ ] **6. 검증 패널 텍스트 오버플로우 수정** — 검증 패널 안내 문구(`confirmed/disputed/debunked/corrected ...`)가 카드 영역을 초과해 잘리는 레이아웃 버그를 고친다(word-break/줄바꿈 처리).
- [ ] **7. `/archive` 승인 데이터 조회** — 검증이 끝나 승인(공개)된 데이터를 일반 사용자가 `/archive`에서 조회하는 기능을 만든다.
  - **진실 공급원**: 승인본은 Cloudflare **D1이 단일 출처**다(검증 큐는 GitHub Issues가 아님 — `steering/github-issues-validation-queue-deprecated.md`).
  - **지금**: intake-api(또는 관리자 API)에서 승인된 데이터를 내려주는 공개 조회 엔드포인트로 **API 연동 방식**으로 구현한다(목록·상세·카테고리 필터 등).
  - **나중(준비만)**: 향후 승인 데이터를 **static build(SSG)로 배포**할 수 있도록 데이터 접근 계층/타입/페이지 구조를 분리해 **준비만 해둔다**(지금 static 빌드까지 하지는 않음). 어떤 경로로 정적 산출물을 만들지(빌드타임 fetch → JSON 산출 등) 스펙에 한 줄 메모로 남긴다.
  - **공개 배포 데이터 정합**: GitHub `3dulev/votatis-data` 레포는 **공개 데이터 배포용**(이슈 본문이 YAML frontmatter 구조: `submission_id/election/title/status/tags/region/collected_at/sources[]/submitter/license`)이다. `/archive` 표시 필드와 카테고리(투개표/사전투표/전산집계 ↔ `tags`)가 이 구조와 어긋나지 않도록 맞춘다.
- [ ] **8. 제보 검토 피드백 스키마 도입** — 제보를 검토(판정)할 때 일관된 YAML 피드백을 남긴다. 스키마는 `Votatis/Votatis#2`(제보 피드백 스키마 초안)를 따른다. 핵심 원칙: ①제보자 주장(`claim`)과 관찰 사실(`verified_facts`)과 검토자 판단(`assessment`) 분리, ②`확인됨`은 `status_scope`로 확인 범위를 반드시 제한(부정선거 단정 금지), ③과잉 해석 방지를 위해 `not_confirmed`를 반드시 기록. 이 원칙들이 페르소나 5(검증자)의 확증편향을 막는 장치이므로 **데이터 모델·검토 UI가 이를 강제**하도록 만든다.
  - **필수/옵셔널 필드 판단(의사결정 원칙에 따라 확정)** — 검증 무결성·중립성을 강제하는 필드를 필수로 둔다.
    - **필수**: `submission_id`, `recommended_status`, `claim`, `verified_facts`(최소 1개), `not_confirmed`(최소 1개 — 과잉해석 차단 장치), `public_summary`, `risk_level`.
    - **조건부 필수**: `recommended_status == "확인됨"`이면 `status_scope` + `confirmed_scope`(최소 1개) 필수. `recommended_status` 값은 스키마의 8종(`확인됨/검토중/미검증/이견있음/반박됨/정정됨/반려/보류`) 중 하나로 제한.
    - **옵셔널(권장)**: `election`(단일 선거 기본값 가능), `recommended_title`, `assessment`, `possible_explanations`(중립성 위해 확인됨/이견있음일 때 강권), `missing_evidence`, `reviewer_note`(내부용), `tags`.
- [ ] **9. 예시 데이터로 end-to-end 테스트 + 삽입** — `3dulev/votatis-data`의 기존 제보(9건)를 **우리 예시 데이터**로 삼아, 제보 입력(항목 4) → 검토 피드백 스키마(항목 8)로 판정 → 승인본이 `/archive`(항목 7)에 노출되는 전체 흐름을 테스트하고, 예시 데이터를 실제로 **D1에 삽입(시드)까지** 해 둔다. 시드 스크립트/마이그레이션 형태로 재현 가능하게 남긴다.

## 작업 방식 (필수) — 스펙 주도 개발

이 묶음은 **스펙 주도 개발 스킬로만 진행한다.** 위 할 일을 임의로 바로 코드부터 고치지 말고, 다음 흐름을 따른다.

1. **스펙 작성 (`spec-create`)**: 위 6개 할 일을 묶을 스펙을 `specs/not-started/`에 작성한다. 묶음이 하나의 스펙으로 과하면 응집도 기준으로 나눈다(예: 관리자 페이지 UX 정리 / 제보 카테고리·입력). 스펙 분할·범위는 의사결정 원칙에 따라 네가 정한다.
2. **구현 (`spec-implement`)**: 스펙을 `in-progress`로 옮겨 코드로 구현한다. 구현·검증(빌드/타입체크/테스트)이 끝나면 `in-review`로 옮긴다.
3. **검토 (`spec-review`)**: 완료 조건 충족을 검토하고 통과 시 `completed`로 옮긴다.
4. 상태 전환은 파일 이동으로 표현하고, `specs/README.md` 인덱스를 즉시 동기화한다.
5. 동작이 바뀌는 변경은 해당 스펙 `## Changelog`에 한 줄 기록(`updated`도 갱신). steering 회상·캡처는 평소 규칙대로.

## 매 실행 루틴 (이 파일을 /goal로 먹였을 때)

1. 진실 공급원과 `specs/README.md`를 읽어 현재 상태 파악(이미 만든 스펙이 있으면 그 단계부터 이어간다).
2. 스펙이 없으면 `spec-create`부터, 있으면 `spec-implement`/`spec-review`로 가장 우선인 항목을 끝까지 진행하고 체크박스·인덱스를 갱신한다.
3. 막히면 사유를 기록하고 다음 가능한 작업으로.
4. 더 진행할 수 없으면 남은 항목을 요약 보고하고 멈춘다.
