# Goal 002 — 관리(검증) 과정용 MCP 서버

## 한 줄 목표

우리 프로젝트의 **관리(검증) 과정에 유용한 MCP 서버**를 기획하고, `docs/PERSONA.md`의 **내부 검증 관리자(페르소나 5, 정한울)**를 1순위로 고려해 설계한 뒤, 기획 문서를 `docs/`에 추가하고 그 기획대로 **MCP를 스펙 주도로 구현·완성**한다. 마지막으로 `docs/mcp-usage.md`를 만들어, 나중에 AI(Claude Code 또는 Codex)에게 먹이면 **바로 이 MCP를 쓸 수 있게** 한다.

## 진실 공급원 (먼저 읽어라)

- `docs/PERSONA.md` — 특히 **페르소나 5(내부 검증 관리자)**: "빠른 통과가 아니라 엄격한 판정을 쉽게", 확증편향 방지, AI는 보조 신호. MCP의 도구 설계·가드레일은 이 사람을 기준으로 한다.
- `docs/MVP-PRD.md` §6 검증 워크플로우 / §11 관리자 페이지 — 검증 단계와 판정 상태(confirmed/disputed/debunked/corrected).
- 기존 관리자 API(`specs/completed/0008-admin-verification-api.md`, `apps/intake-api/src/routes/admin.ts`): 검토 큐·상세·판정(PATCH, 근거 필수)·증거 스트리밍·보조 분석(analyze)·export·stats. **MCP는 이 API를 감싸는 얇은 계층이어야 한다**(로직 중복 금지).
- `CLAUDE.md` — 모노레포(`apps/<name>/`), 스펙 주도 개발, steering, Changelog 정책.
- `loops/HUMAN.md` — 사람만 할 수 있는 일을 적는 곳.

## 작업 방식 (필수)

1. **기획 먼저(docs/)**: MCP의 목적·대상 사용자(페르소나 5)·노출할 도구 목록·각 도구의 입출력·가드레일(판정 시 근거 필수 등)·전송 방식(stdio vs 원격)·인증(ADMIN_TOKEN)·아키텍처를 `docs/`에 기획 문서로 작성한다(예: `docs/mcp-admin-plan.md`). 이름·구성은 PERSONA 기준으로 네가 정한다.
2. **스펙 주도 구현**: `spec-create`로 스펙 작성 → `spec-implement`로 구현(상태 이동) → `spec-review`로 검증 후 `completed`. 새 앱은 `apps/<name>/`에 둔다(루트 금지).
3. **얇은 래퍼 원칙**: 검증 로직(근거 필수, 상태 전이 규칙 등)은 intake-api가 단일 출처. MCP는 그 API를 호출만 한다. 도구 설명(description)·스키마로 AI가 올바르게 쓰도록 유도한다.
4. **사용 가이드**: `docs/mcp-usage.md` — Claude Code(`.mcp.json`/`claude mcp add`)와 Codex(`~/.codex/config.toml` 등) 양쪽에서 이 MCP를 등록·인증·사용하는 법, 도구 목록, 전형적 검증 시나리오(큐 확인→증거·분석 검토→근거와 함께 판정)를 담아, 그대로 복붙·먹이면 작동하게 한다.
5. **steering 회상·캡처**, **Changelog**: 평소 규칙대로.
6. **검증**: 빌드/타입체크/테스트가 통과해야 하고, MCP 서버가 실제로 기동되어 도구 목록을 노출하는지(가능한 범위에서) 확인한다.

## 의사결정 원칙 (가장 중요)

**모든 제품·기술·설계 결정은 네가 직접 내린다.** 사람에게 방향을 되묻지 않는다. 판단 기준은 하나:

> **"이 결정이 `docs/PERSONA.md`의 사용자, 특히 내부 검증 관리자(페르소나 5)에게 유익한가?"**

- 도구를 무엇을 노출/숨길지, 가드레일을 어디까지 둘지, 도구 description을 어떻게 쓸지 등은 — **검증자가 더 엄격하고 일관되게(확증편향 없이) 판정하도록 돕는** 쪽으로 정한다.
- AI 보조 신호는 "참고"로만 제시하고 자동 판정하지 않는다(페르소나 5 규율).
- 결정 근거(어느 페르소나에게 왜 유익한지)를 스펙/커밋/Changelog에 한 줄로 남긴다.
